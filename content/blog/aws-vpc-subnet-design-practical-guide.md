## はじめに

AWS の VPC を設計するとき、「サブネットをどう切るか」は必ず議論になります。

よくある失敗は、サブネットを **細かく切りすぎる** ことです。サービスごと、用途ごと、レイヤーごとに細かく分割した結果、サブネット数が膨らみ、管理が煩雑になり、IP アドレスも無駄に消費する — という状態に陥りがちです。

この記事では、本番環境でサブネットを再設計した経験をもとに、VPC の CIDR 設計とサブネットの切り方について整理します。

## VPC の CIDR は /16 で作っておく

VPC を新規に作るなら、CIDR は **`/16` で確保しておく**のがおすすめです。

**後から広げるのは意外と面倒**

VPC の CIDR は後からセカンダリ CIDR を追加できます。VPC 内のローカルルートは自動で追加されるため、VPC 内部だけなら影響は限定的です。ただし、VPC ピアリングや Transit Gateway、VPN 接続がある場合は対向側のルーティング変更が必要になり、接続先が多いほど面倒になります。

**サブネットを大きく切れる**

`/16` なら 65,536 個の IP アドレスが使えます。サブネットを `/20`（約 4,000 IP）で切っても 16 個作れるので、余裕のある設計ができます。`/24` で VPC を作ると 256 IP しかなく、サブネットを複数切るだけで枯渇が見えてきます。ECS Fargate はタスクごとに IP を 1 つ消費するため、小さいサブネットではスケールアウト時に詰まります。

**コストがかからない**

CIDR の割り当てに料金は発生しません。`/16` でも `/24` でもコストは同じです。大きく取るデメリットは他の VPC やオンプレとの CIDR 重複リスクくらいで、マルチアカウント環境では [VPC IPAM](https://docs.aws.amazon.com/ja_jp/vpc/latest/ipam/what-it-is-ipam.html) で管理すれば回避しやすくなります。

## サブネットを分ける基準は「ルートテーブル」

サブネットを分けるかどうかを考えるとき、意識したいのがこの原則です。

**ルートテーブルの要件が異なるなら分ける。同じなら、基本的には分けなくてよい。**

サブネットはルートテーブルと NACL を適用する単位です。同じルートテーブル・同じ NACL を使うリソースであれば、別々のサブネットに分ける必然性は薄くなります。リソース間のアクセス制御はセキュリティグループで対応できることがほとんどです。

### ルートテーブルが異なる典型例

| パターン | ルーティング |
|--|--|
| Public | `0.0.0.0/0` → IGW |
| Private | `0.0.0.0/0` → NAT Gateway |
| Isolated | デフォルトルートなし |

多くの環境では、この 3 種類がベースになります。

### 分けなくてよいケース

- **App と DB** — どちらも NAT 経由のルートテーブルなら、同じ Private サブネットで運用できる。DB へのアクセス制限はセキュリティグループで制御可能
- **サービス A とサービス B** — アウトバウンド要件が同じならサブネットを分ける必然性は低い
- **CI/CD** — App と同じルートテーブルなら独立させる理由がない

ただし、コンプライアンス要件で NACL による分離が求められる場合など、ルートテーブル以外の理由で分けることもあります。一律に「分けるな」ではなく、**分ける理由を説明できるかどうか**で判断するのがポイントです。

## サブネットは大きく切る

`/16` VPC なら、各サブネットは `/20`（約 4,000 IP）程度で切るのがおすすめです。

「IP がもったいない」と感じるかもしれませんが、使わない IP にコストはかかりません。一方、サブネットが小さすぎて IP が枯渇すると、サブネットの作り直しとリソースの再配置が必要になります。後から広げるコストを考えると、最初から余裕を持たせておく方が安くつきます。

ECS Fargate の `awsvpc` モードではタスクごとに ENI が割り当てられるため、`/24`（約 250 IP）だとバッチ処理のスケールアウトですぐに限界がきます。`/20` なら約 4,000 IP で、まず問題にはなりません。

## 設計例

以下は 3 AZ 構成の設計例です。ルートテーブルの要件が異なるもののみサブネットを分けています。

```
VPC: 10.0.0.0/16

Public（IGW 直通）
  prod-public-1a: 10.0.0.0/20
  prod-public-1c: 10.0.16.0/20
  prod-public-1d: 10.0.32.0/20
  → ALB, NAT Gateway

Private（NAT 経由）
  prod-private-1a: 10.0.48.0/20
  prod-private-1c: 10.0.64.0/20
  prod-private-1d: 10.0.80.0/20
  → ECS, EC2, Lambda, RDS, ElastiCache

Isolated（外部経路なし）
  prod-isolated-1a: 10.0.96.0/24
  prod-isolated-1c: 10.0.97.0/24
  prod-isolated-1d: 10.0.98.0/24
  → PrivateLink エンドポイント
```

9 サブネットでシンプルに構成しています。`/16` VPC で `/20` サブネットを使っても空き CIDR は大量に残るため、将来 VPN 用サブネットなどが必要になったときに追加すれば十分です。

### アンチパターン: 細かく切った設計

```
prod-public-app-a-1a: /26
prod-public-app-b-1a: /26
prod-private-app-a-1a: /22
prod-private-app-b-1a: /22
prod-private-db-a-1a: /26
prod-private-db-b-1a: /26
prod-private-cicd-1a: /27
prod-private-endpoint-1a: /26
...（x 3 AZ）
```

サブネット数が 20 個以上に膨らみますが、ルートテーブルの要件は同じものがほとんどです。管理対象が増えるだけで、得られるメリットは限定的です。

## まとめ

- VPC の CIDR は **`/16` で作る**。コストはかからず、後から広げるのは面倒
- サブネットを分ける基準は **ルートテーブルの要件が異なるかどうか**
- 多くの環境では **Public / Private / Isolated の 3 種類** がベース
- サブネットは **大きく切る**。`/20` 以上がおすすめ。IP 枯渇で作り直す方がはるかに高くつく
- 分ける理由を説明できないサブネット分割は、管理コストを増やすだけになりがち
