## ファクトチェック

ブログ記事（`content/blog/*.md`）の内容を検証し、事実誤認や古い情報がないかチェックする。

### 使い方

引数にファイル名（スラッグ）を指定する。省略時は全記事を対象にする。

```
/fact-check azure-mysql-80-to-84-upgrade
/fact-check          # 全記事
```

### チェック項目

1. **技術的正確性**: サービス名・機能名・バージョン番号・コマンド・設定値が正しいか。公式ドキュメントと照合する。
2. **日付・期限**: サポート終了日、リリース日、移行期限などが最新情報と一致しているか。Web検索で裏取りする。
3. **用語の一貫性**: 同一概念に対して表記揺れがないか（例: "Flexible Server" と "フレキシブルサーバー" の混在）。
4. **数値・ベンチマーク**: 具体的な数値（レイテンシ、コスト、スループット等）に根拠があるか、桁が妥当か。
5. **リンク・参照**: 記事内で言及している外部リソース（公式ドキュメント等）のURLが存在し、内容と一致しているか。
6. **frontmatter**: `title`, `date`, `description`, `tags` が適切か。descriptionが記事内容を正確に要約しているか。

### 手順

1. 対象の記事ファイルを読み込む
2. 記事内の事実主張（日付、バージョン、サービス仕様など）をリストアップする
3. 各主張について Web 検索および公式ドキュメントの確認で裏取りする
4. 以下のフォーマットでレポートを出力する

### 出力フォーマット

```
## ファクトチェック結果: {記事タイトル}

### 検証済み（問題なし）
- ✅ {主張} — {根拠・出典}

### 要確認
- ⚠️ {主張} — {懸念点・最新情報との差分}

### 誤り
- ❌ {主張} — {正しい情報と出典}

### 用語・表記
- {指摘事項}

### 総合判定
{記事全体の信頼性についてのまとめ}
```

### 注意事項

- 検索結果の日付に注意し、記事の公開日時点で正しかった情報と、現在は古くなった情報を区別すること
- 不確実な場合は「要確認」に分類し、断定的に誤りとしない
- 記事の主張だけでなく、暗黙の前提（例: 特定リージョンでの可用性）も検証対象とする
